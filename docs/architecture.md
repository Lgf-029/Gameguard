# GameGuard 架构设计文档

## 一、系统概述

GameGuard 是一个面向游戏运营的多智能体风控检测系统，按生产级标准设计。系统接收交易请求后，通过 LangGraph 编排三路并行检测（规则引擎 + 向量相似度 + 图关系异常），经 RRF 融合输出综合风险分，根据阈值自动决定拦截、人工审核或放行。

## 二、技术架构总览

POST /detect 进来后，FastAPI 入口将请求交给 LangGraph 编排层。编排层通过 Send 同时分发三路检测——规则引擎、向量检测、图检测——三路并行执行，完成后汇聚到 RRF 融合节点，根据融合分数做出最终风险决策（拦截/人工审核/放行）。

架构示意图：

```
POST /detect
    │
    ▼
┌───────────────┐
│  FastAPI 入口  │
└───────┬───────┘
        │
        ▼
```

## 三、五库联动协同逻辑

MySQL 存储规则库和玩家/设备/交易等业务数据，规则引擎从这里加载规则定义并查询玩家画像和设备信息。

Redis 承担规则缓存和滑动窗口计数，热点规则 600 秒缓存，ZSet 记录 1 小时内充值次数。

Milvus 存储异常模式向量和交易行为向量，向量检测将当前交易向量与历史异常向量做余弦相似度匹配。

Neo4j 存储设备-账号图关系，图检测查询设备关联账号数，识别设备农场和共享设备异常。

Elasticsearch 存储检测日志，每笔检测结果写入 ES，支持按玩家、时间、风险等级查询。

## 四、检测链路设计

规则引擎加载 MySQL 规则定义（8 条规则，enabled=1 启用），查询玩家画像（注册天数、设备数、IP 数、累计充值），查询设备信息（是否模拟器、是否 ROOT、关联账号数），Redis 滑动窗口统计 1 小时内充值频率，逐条匹配规则，输出触发规则列表和规则总分。

向量检测将交易信息（uid、金额、设备、IP、支付方式）拼接为行为描述文本，通过 DashScope text-embedding-v4 生成 1024 维 Embedding，在 Milvus 的 transaction_behaviors 集合中检索 Top-5 最相似历史异常交易，将余弦距离映射为相似度分数（1 - distance），输出相似度分和匹配列表。

图检测查询 Neo4j 中当前设备关联的账号数，设备关联 5 个以上账号判定为设备农场异常（分数为 account_count × 12，上限 60），同时查询当前玩家与其他玩家共享的设备，共享设备关联 3 个以上其他账号判定为共享设备异常（分数为 shared_count × 15，上限 45）。

## 五、RRF 融合与决策路由

三路检测输出不同尺度的分数（规则分 0-100、向量相似度 0-1、图异常分 0-100），取平均值后映射风险等级：

| 风险分 | 等级 | 处置 |
|--------|------|------|
| ≥80 | high | 自动拦截 |
| 40-79 | medium | 人工审核 |
| <40 | low | 放行 |

## 六、关键设计决策

- Redis ZSet 滑动窗口：替代 INCR 方案，每条事件独立时间戳，避免 TTL 边界绕过攻击
- Embedding API 调用：本地 16GB 内存无法运行 BGE 模型，选 DashScope API，入行后按延迟需求切换本地模型
- 检测器模块级懒加载单例：每个检测器文件独立管理数据库连接，首次调用建连、后续复用
- Send 三路并行：规则引擎、向量检测、图检测互相独立，从 START 直接分发，形成有向无环图，避免循环调用
